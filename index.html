01 — 문제 정의

영어 말하기, 왜 여전히 어려운가

안전하게 반복 연습할 공간이 없다. MEF는 비용·심리적 부담·피드백 부재를 동시에 해결한다.

🧊

실전 경험 부족

단어와 문법을 알아도 실제 대화 상황이 오면 입이 얼어버린다. 일상에서 영어를 쓸 기회가 없다.

→ 언제든 원하는 상황으로 연습 가능

💸

높은 접근 비용

원어민 화상 영어는 1회에 3~10만원. 꾸준히 이어가려면 한 달에 수십만 원이 필요하다.

→ AI 모델로 비용 장벽 완전 제거

😰

심리적 부담

실수에 대한 두려움, 발음에 대한 수치심. 사람 앞에서 영어를 쓰는 것 자체가 스트레스다.

→ AI 상대라 부담 없이 반복 연습 가능

🔁

피드백 부재

혼자 연습하면 같은 실수를 반복하게 된다. 잘못된 표현이 굳어버리는 악순환이 이어진다.

→ 대화 후 문법·표현·점수 피드백 즉시 제공

03 — 솔루션

4개 AI Agent로 어떻게 해결하는가

"상황 입력 하나로 시나리오 → 대화 → 피드백 → 분석까지 자동"

🎬 ScenarioAgent

이동희 담당 · scenario_service.dart

한국어 상황 → AI가 배경·역할·목표를 JSON으로 자동 생성. responseMimeType 강제 적용.

입력: "카페 주문" → 바리스타 역할 시나리오

💬 ConversationAgent

김령균 담당 · conversation_service.dart

시나리오 기반 실시간 영어 대화. 이전 대화 전체를 API에 전달해 맥락 유지.

AI는 역할에 맞게 영어로만 응답

📋 FeedbackAgent

김령균 담당 · feedback_service.dart

대화 종료 후 문법 오류, 자연스러운 표현, 점수(100점)를 JSON으로 제공.

예: "I am go" → "I go" / 점수: 78점

📊 AnalysisAgent

이동희+김령균 · analysis_service.dart

누적 대화 기록을 분석해 반복 실수 패턴을 개인화 리포트로 제공.

예: "3인칭 단수 's' 누락"이 70% 반복

04 — 프로젝트 진행 과정

어떻게 만들었나 — 시행착오와 결정

설계 → 환경 구축 → 기능 구현 → 통합 순서로 진행. 팀원 A·B 병렬 개발 후 main 브랜치 병합.

✓

아키텍처 결정 — Supabase 제거

시행착오: 초기에 Supabase 백엔드를 계획했으나, MVP 우선 원칙으로 서버 없이 직접 API 호출로 전환. 인증은 메모리 더미(test@mef.com)로 대체. → ADR-0003, ADR-0004

✓

AI API 변경 — Claude → OpenRouter → Gemini 직접 호출

시행착오: Claude API 과금 이슈 → OpenRouter 무료 모델 → 최종적으로 Google Gemini 2.5 Flash 직접 호출로 결정. → ADR-0005

✓

JSON 파싱 오류 수정 — responseMimeType 도입

시행착오: Gemini 2.5 Flash(thinking 모델)가 JSON 외 텍스트를 붙이는 경우 발생 → responseMimeType: 'application/json' 강제 설정으로 해결.

✓

팀 병합 충돌 해결 — auth_provider, app_router 충돌

시행착오: A·B가 같은 파일을 수정해 merge conflict 발생 → 팀원 B 버전 채택 후 A 기능 통합. /signup vs /sign-up 경로 통일.

07 — 앱 구조

새 화면을 추가하면 어느 폴더에 만드는가

Layered Architecture: screens → providers → services → models 단방향 의존. 화면에서 API 직접 호출 금지.

lib/ 디렉토리 구조

├── screens/ ← 화면 (UI만)
│ ├── auth/ 로그인·회원가입
│ ├── scenario/ 시나리오 입력·결과
│ ├── conversation/ 채팅
│ ├── feedback/ 피드백
│ ├── history/ 대화 기록
│ └── analysis/ 약점 분석
├── providers/ ← 상태 관리 (Riverpod)
├── services/ ← AI API 호출 전담
├── models/ ← 데이터 구조
├── constants/ API 키 (gitignore)
└── router/ go_router 설정

레이어 책임 원칙

Gemini API 호출 전담. 프롬프트 + JSON 파싱.

models

Scenario, Feedback, AnalysisReport, Message 구조.

11 — 성능 최적화

성능 최적화를 위해 어떤 노력을 했나

API 응답 안정성, 토큰 효율, 에러 처리 3가지 축으로 최적화 진행.

🎯 JSON 파싱 안정화

문제: Gemini 2.5 Flash(thinking 모델)가 JSON 외 텍스트를 포함해 파싱 실패 발생

해결: generationConfig에 responseMimeType: 'application/json' 강제 설정 → 파싱 오류 0%

💰 토큰 최적화

문제: maxOutputTokens 미설정 시 불필요한 토큰 소비 → API 비용·응답 속도 악화

해결: 용도별 차등 설정 — Scenario 1024 / Conversation 512 / Feedback 2048 / Analysis 4096

⚡ 상태 관리 최적화

문제: API 호출 중 UI 블로킹 → 사용자가 앱이 멈춘 것으로 오해

해결: AsyncValue의 loading/data/error 3상태로 로딩 인디케이터 자동 처리. 화면 블로킹 없음.

🛡 에러 처리 강화

문제: 429 Too Many Requests, 네트워크 오류 시 앱 크래시

해결: 각 서비스에 statusCode 체크 + try/catch fallback 응답. 사용자에게 친화적 메시지 표시.

지속적 개선 의지: 현재 API 키가 클라이언트에 노출됨 → 실서비스 전환 시 서버 프록시(Edge Function) 도입. 대화 기록 영속화(SharedPreferences) 추가 예정.
