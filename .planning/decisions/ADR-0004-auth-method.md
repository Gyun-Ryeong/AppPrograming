# ADR-0004: 인증 방식 — Mock 메모리 인증

- 상태: Accepted
- 날짜: 2026-05-24
- 결정자: MEF 팀

## 배경

MEF 앱은 사용자 로그인/회원가입 기능이 필요하다.
초기에는 Supabase Auth를 사용할 계획이었으나, Supabase 제거(ADR-0003) 결정으로
대체 인증 방식이 필요해졌다.

## 고려한 대안

### 대안 A: Supabase Auth
- 장점: 실제 이메일 인증, 세션 유지, 보안
- 단점: Supabase 제거로 사용 불가 (ADR-0003)

### 대안 B: Firebase Auth
- 장점: Google 계정 연동, 실제 인증
- 단점: Firebase 별도 셋업 필요, 7주 일정에 부담

### 대안 C: Mock 메모리 인증 (선택)
- 장점: 셋업 시간 0, 즉시 구현 가능, 발표 데모용으로 충분
- 단점: 앱 종료 시 계정 초기화, 실제 보안 없음

## 결정

**Mock 메모리 인증**을 선택한다.
Riverpod 메모리 Map에 사용자 계정을 저장하고, 앱 실행 중에만 유효하다.

## 이유

1. **7주 제약**: 실제 인증 구현보다 핵심 기능(시나리오, 대화, 피드백)에 집중
2. **수업 과제 목적**: 인증 자체가 평가 대상이 아님
3. **데모 충분성**: 발표 시 `test@mef.com / test1234`로 즉시 로그인 가능

## 결과

긍정:
- 로그인/회원가입 UI 완성 (실제 동작)
- 발표 데모에서 인증 흐름 시연 가능

부정 / 제약:
- 앱 종료 시 모든 계정 초기화
- 실제 서비스 전환 시 Supabase Auth 또는 Firebase Auth 도입 필요

## 발표 Q&A 대비

> "인증은 어떻게 구현했나요?"
→ "수업 과제 기간 내 핵심 기능 구현에 집중하기 위해 메모리 기반 Mock 인증을 사용했습니다.
   실제 서비스 전환 시에는 Supabase Auth를 도입할 계획입니다."

## 후속 작업

- [ ] 실제 서비스 전환 시 Supabase Auth 또는 Firebase Auth 도입
