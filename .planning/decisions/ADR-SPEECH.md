> 각 ADR당 60초 안에 말할 수 있도록 연습할 것.

## ADR-0001: Flutter 선택

> "우리 팀은 **Flutter**를 선택했습니다.
> 대안으로는 **React Native**가 있었지만,
> Flutter는 자체 렌더링 엔진을 사용해 iOS와 Android에서 **동일한 UI**를 보장하고,
> JavaScript Bridge 없이 Dart가 네이티브로 컴파일되어 **성능이 더 안정적**입니다.
> 또한 7주라는 짧은 기간 안에 양쪽 플랫폼을 동시에 개발해야 했기 때문에
> 코드 재사용성이 높은 Flutter가 더 적합하다고 판단했습니다.
> 이 결정은 ADR-0001에 기록되어 있습니다."

**핵심 키워드:** 자체 렌더링 엔진 / JS Bridge 없음 / 단일 코드베이스 / 7주 단기 개발

## ADR-0002: Riverpod 선택 (상태 관리)

> "우리 팀은 상태 관리 라이브러리로 **Riverpod**을 선택했습니다.
> 대안으로는 **Provider**와 **BLoC**이 있었지만,
> Provider는 context 의존성 문제가 있고,
> BLoC은 보일러플레이트가 많아 7주 과제에는 과도한 복잡도였습니다.
> Riverpod은 컴파일 타임에 잘못된 상태 접근을 잡아주고,
> context 없이도 어디서든 상태에 접근할 수 있어
> services/ 레이어 분리 원칙과도 잘 맞았습니다.
> 이 결정은 ADR-0002에 기록되어 있습니다."

**핵심 키워드:** 컴파일 타임 안전성 / context 독립 / Provider 개선판 / 레이어 분리

---

## ADR-0003: Supabase 선택 (백엔드)

> "우리 팀은 백엔드로 **Supabase**를 선택했습니다.
> 대안으로는 **Firebase**와 **직접 서버 구축**이 있었지만,
> Firebase는 NoSQL 구조라 세션-메시지-피드백 간의 관계형 데이터를 표현하기 불편했고,
> 직접 서버는 7주 안에 완성하기 어려운 범위였습니다.
> Supabase는 PostgreSQL 기반으로 관계형 DB를 쓸 수 있고,
> Edge Function으로 Anthropic API 키를 서버에서만 관리할 수 있어
> 클라이언트에 키가 노출되지 않습니다.
> 이 결정은 ADR-0003에 기록되어 있습니다."

**핵심 키워드:** PostgreSQL 관계형 DB / API 키 보호 / Edge Function / 무료 티어
